# VEX-style engineering notebook (Google Slides)

Use this checklist so entries read like **evidence-backed engineering work**, not marketing slides. Align with common VEX design-process expectations: problem → brainstorm/constraints → selection → build → test → iterate → next steps.

## What graders and judges usually look for

- **Clear problem / goal** tied to the game or assignment (mechanism purpose, constraints: size, motor ports, time).
- **Design exploration**: multiple ideas or iterations with a **stated reason** the team picked one path (tradeoffs, risk, simplicity, reliability).
- **Build documentation**: how parts went together, key assembly decisions, CAD/hand-sketch references if your class uses them.
- **Test plan and results**: what you tested, how you measured, what happened (pass/fail, surprising behavior). Avoid inventing data—mark “TBD” until measured.
- **Iteration log**: what failed or underperformed, what you changed, why.
- **Team process** (if required): roles, meeting notes, deadlines—only when your teacher’s rubric asks for it.

## Slide-friendly patterns

- One slide = one idea (problem, test, iteration, or decision). Prefer bullets, short captions, labeled photos.
- **Figure captions** should answer: what is shown, why it matters, what changed from last time.
- **Tables** for test matrices (trial, input, outcome) when appropriate.

## Terms to use correctly

- Distinguish **assumption** vs **observation** vs **decision**. Do not present guesses as measurements.

## Anti-patterns to avoid

- Generic fluff (“we worked hard”, “it is very good”) without evidence.
- Long paragraphs where a diagram + caption would suffice.
- Copy-pasting vague game descriptions without tying back to *your* robot decisions.
